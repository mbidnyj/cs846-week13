# Original Requirements: Microblogging Web Application (Twitter-like)

### Features:
- Create a user profile
- Post short text updates (limited length)
- View a chronological feed of posts from all users
- Like posts
- Reply to posts (one level deep)
- Login to user profile
- View a user’s profile and their posts

### Constraints:
- No private messaging
- No retweets / reposts
- No follower graph (global feed only)

# Microblogging Web Application: Technical Planning Questions

## 1. User Authentication and Profile Management
- Authentication will use a simple email and password form. User credentials will be stored in memory on the backend server (no persistent storage, no encryption).
- User sessions will be managed in the simplest way: a boolean flag indicating if a user is logged in or not, stored in server memory (no JWT, cookies, or session storage).
- User profile fields: username, email, password, and bio. No avatar or additional fields.
- Profile updates will be handled by sending the updated data to the backend, which will overwrite the in-memory user object. On page reload, the frontend fetches the latest data from the server.
- Passwords will be stored as plain text in memory (no hashing or security, as this is a POC only). No password reset functionality.
- No restrictions or validation on usernames or profile data beyond what is required for the in-memory structure.

## 2. Posting and Storing Short Text Updates
- Maximum post length: No enforced limit; keep it simple for the POC.
- Posts storage: All posts are stored in a Python in-memory data structure (e.g., a list or dictionary) on the backend server. No database is used.
- Post formatting: Posts are plain text only; no markdown or rich formatting supported.
- Spam/abuse handling: No spam or abuse detection implemented for this prototype.
- Edit/delete: Users can edit or delete their own posts. Edit and delete operations simply update or remove the post from the in-memory structure.

## 3. Feed Generation and Display
- The global feed will be generated by simply returning all posts from the in-memory data structure on the backend, ordered by creation time.
- Pagination will be handled by returning a fixed number of posts per request (e.g., 10 per page), with simple offset/limit parameters in the API.
- Sorting will be strictly chronological (newest first), as this is the simplest approach.
- The frontend will fetch posts from all users via a REST API endpoint (e.g., /api/feed), which returns the paginated list of posts.
- No caching or optimization will be implemented for feed loading; every request fetches the latest data from memory.
- For large numbers of posts, only a limited number (e.g., 10) will be returned per API call to avoid overloading the frontend.


## 4. Like and Reply Functionality
Likes will be stored as a set of user IDs per post in the in-memory data structure on the backend. Each post object will have a list or set of user IDs who liked it. No sorting by likes is required.

Users will not be able to unlike posts; liking is a one-way action for this prototype.

Replies will be stored as a list of reply objects attached to each post in the in-memory structure. Each reply will have a reference to the parent post, the author, and the reply text. Only one level of replies is supported (no replies to replies).

Replies will be displayed directly under the parent post in the UI, indented or visually grouped. This is the most common and simple approach for one-level replies.

Users will not be able to delete their replies.

## 5. User Profile Viewing
- The user's profile page will display only the bio and a simple status (if any), along with a list of their posts.
- When viewing a profile, the frontend will fetch the user's posts and profile data from the backend's in-memory storage each time the page is loaded or refreshed.
- There will be no privacy settings; all profiles and posts are public and visible to everyone.

## 6. Application Architecture
- Monolithic architecture is best for this simple POC. All backend logic will be in a single Python server process.

- The backend will handle all data storage and business logic in memory. The frontend will fetch and display data via API calls.

- API endpoints will be created only for the required features (login, register, post, feed, like, reply, profile, etc.). Documentation will be minimal and inline in the code.

- REST API will be used, as it is the simplest and most common approach for this type of application.

## 7. Technology Stack Choices
Backend: Python. Use Flask for the backend framework, as it is lightweight and easy to set up for a simple REST API.

Database: No database will be used; all data is stored in memory on the backend server.

Third-party services: No third-party services (authentication, storage, etc.) will be used.

Frontend: Use plain HTML/CSS/JavaScript for the frontend, or optionally use React if a component-based approach is desired. For maximum simplicity, plain JS is recommended.

## 8. Security, Scalability, and Deployment

- No protection against common web vulnerabilities (XSS, CSRF, etc.) will be implemented, as this is a local POC.
- User data will not be secured in transit or at rest; all data is in server memory only.
- No scaling plan; the application is intended for local, single-user or small group testing only.
- Deployment will be local only (run Flask server and open frontend in browser).
- Logging will be done using simple print statements in the backend for major actions (user login, post creation, etc.). No monitoring tools will be used.
